Using Webpack With TypeScript

In this section I learned how to use webpack to bundle code. This is especially advantageous whenever a project grows and our files start to become large. This reduces the loading times of our web applications because we now only need to fetch a single optimized minified file.

Setting up the entry point and output

We begin by setting up the entry point(first point of contact for the application logic) where webpack will begin bundling from and the output where the bundled file will be created. To do this we first create a file named webpack.config.js in the root folder that will contain all configurations for webpack. Here we export an object with the various configuration options. We also need to modify the tsconfig.json file where we comment out the "rootDir" configuration since we no longer need because webpack will handle the compilation of files. 

  1. // webpack.config.js
  2. const path = require('path');
  3.  
  4. module.exports = {
  5.   entry: './src/app.ts',
  6.   output: {
  7.     filename: 'bundle.js',
  8.     path: path.resolve(__dirname, 'dist')
  9.   },
  10.   // other configs...
  11. };
Configuring loaders(ts-loader)

Loaders are configured using the module.rules field where webpack will evaluate and execute the loaders using a right to left or bottom to top approach. These loaders are generally packages that help in the compilation of files. 

  1. // webpack.config.js
  2. const path = require('path');
  3.  
  4. module.exports = {
  5.   entry: './src/app.ts',
  6.   output: {
  7.     filename: 'bundle.js',
  8.     path: path.resolve(__dirname, 'dist')
  9.   },
  10.   module: {
  11.     /**
  12.      * Allows us to specify several loaders within our webpack config.
  13.      * They are evaluated right to left or bottom to top
  14.      * 
  15.      * @link https://webpack.js.org/configuration/module/#modulerules
  16.      */
  17.     rules: [{
  18.       test: /\.ts$/,
  19.       use: 'ts-loader',
  20.       exclude: /node_modules/
  21.     }]
  22.   },
  23.   // other configs...
  24. };
Resolving Modules

Because we are using modules we need to tell webpack which type of files to look for in our import statements. This will allow us to exclude file extensions when writing import statements in files. We do this by configuring the resolve.extensions configuration option. This is passed as an array of strings. 

  1. // webpack.config.js
  2. const path = require('path');
  3.  
  4. module.exports = {
  5.   entry: './src/app.ts',
  6.   output: {
  7.     filename: 'bundle.js',
  8.     path: path.resolve(__dirname, 'dist')
  9.   },
  10.   module: {
  11.     /**
  12.      * Allows us to specify several loaders within our webpack config.
  13.      * They are evaluated right to left or bottom to top
  14.      * 
  15.      * @link https://webpack.js.org/configuration/module/#modulerules
  16.      */
  17.     rules: [{
  18.       test: /\.ts$/,
  19.       use: 'ts-loader',
  20.       exclude: /node_modules/
  21.     }]
  22.   },
  23.   // other configs...
  24. };
Configuring a dev server and build process

Here we use the webpack-cli dependency to build files for production and run a dev server using npm scripts. To spin up a dev server we first need to configure the port and other settings. This is done by the devServer option where we pass configurations such as a public path and port on which to serve. We also configure the devtool option to tell webpack how to handle source maps for use during debugging in development. 

  1. // webpack.config.js
  2. const path = require('path');
  3.  
  4. module.exports = {
  5.   devServer: {
  6.     publicPath: '/dist/',
  7.     port: 8090,
  8.   },
  9.   devtool: 'inline-source-map',
  10.   // other configs...
  11. };

For the npm scripts, we modify the package.json file where we pass a script for the build process and another for the dev server. 

  1. {
  2.   "scripts": {
  3.     "serve": "webpack serve",
  4.     "build": "webpack"
  5.   }
  6. }
Configuring for production and development

Here we are using environment variables to set the mode configuration on webpack. This tells webpack to set the respective optimization options for the environment the application is running in. For this we can use a function export for the configuration options where we check the environment and pass the relevant options. We also set the environment through the npm scripts created earlier. 

  1. const path = require('path');
  2.  
  3. const config = {
  4.   mode: process.env.NODE_ENV,
  5.   entry: './src/app.ts',
  6.   output: {
  7.     filename: 'bundle.js',
  8.     path: path.resolve(__dirname, 'dist')
  9.   },
  10.   module: {
  11.     /**
  12.      * Allows us to specify several loaders within our webpack config.
  13.      * They are evaluated right to left or bottom to top
  14.      * 
  15.      * @link https://webpack.js.org/configuration/module/#modulerules
  16.      */
  17.     rules: [{
  18.       test: /\.ts$/,
  19.       use: 'ts-loader',
  20.       exclude: /node_modules/
  21.     }]
  22.   },
  23.   resolve: {
  24.     /**
  25.      * Tells webpack what files with the given extensions it should work with.
  26.      * Allows us to do file imports without passing the file extensions.
  27.      * 
  28.      * @link https://webpack.js.org/configuration/resolve/#resolveextensions
  29.      */
  30.     extensions: ['.ts', '.js']
  31.   }
  32. };
  33.  
  34. module.exports = (env, argv) => {
  35.   if (process.env.NODE_ENV === 'development') {
  36.     config.output.publicPath = 'dist';
  37.     config.devServer = {
  38.       publicPath: '/dist/',
  39.       port: 8090,
  40.     };
  41.     /**
  42.      * Controls how source maps are generated. And if they exist
  43.      * how they will be bundled.
  44.      * 
  45.      * @link https://webpack.js.org/configuration/devtool/#devtool
  46.      */
  47.     config.devtool = 'inline-source-map';
  48.   }
  49.  
  50.   if (process.env.NODE_ENV === 'production') {
  51.     config.devtool = false;
  52.   }
  53.  
  54.   return config;
  55. };

With the above setup, we just need to set an environment via npm scripts. For this we can use the --node-env flag like so: 

  1. {
  2.   "scripts": {
  3.     "serve": "webpack serve --node-env development",
  4.     "build": "webpack --node-env production"
  5.   }
  6. }
Clean up during production builds

This is done using a dependency called clean-webpack-plugin that is configured to first clear all previous builds in the output directory. This makes sure that we always have the latest build for our logic. 

  1. const path = require('path');
  2. const {
  3.   CleanWebpackPlugin
  4. } = require('clean-webpack-plugin');
  5.  
  6. const config = {
  7.   mode: process.env.NODE_ENV,
  8.   entry: './src/app.ts',
  9.   output: {
  10.     filename: 'bundle.js',
  11.     path: path.resolve(__dirname, 'dist')
  12.   },
  13.   // other configs...
  14. };
  15.  
  16. module.exports = (env, argv) => {
  17.   if (process.env.NODE_ENV === 'development') {
  18.     // do whatever...
  19.   }
  20.  
  21.   if (process.env.NODE_ENV === 'production') {
  22.     config.devtool = false;
  23.     config.plugins = [
  24.       /**
  25.        * This plugin cleans up the output path before doing a re-build in production.
  26.        * 
  27.        * @link https://www.npmjs.com/package/clean-webpack-plugin
  28.        */
  29.       new CleanWebpackPlugin()
  30.     ];
  31.   }
  32.  
  33.   return config;
  34. };
Try it

To run the app, in your terminal run the following commands: 

  1. cd Lesson\ 09\ -\ Webpack\|TypeScript/app/
  2.  
  3. npm install // if you have not installed dependencies
  4.  
  5. npm run compile:type // compiles typescript to common js
  6.  
  7. npm run serve // starts application
